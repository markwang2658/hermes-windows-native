# Storytelling for Technical Leaders

## Trigger Conditions
- User asks for stories, entertainment, or creative narratives
- User requests humor or light-hearted content
- User wants to relax or be entertained

## Core Principles
1. **高强度冲突优先**：用户不喜欢轻微幽默，需要激烈的正面冲突、权力斗争、技术对抗
2. **角色性格鲜明强势**：主角要有严厉将军风格，命令式语气，零容忍低级错误
3. **戏剧性对抗**：场景要有火药味，包括骂战、拍桌子、正面硬刚
4. **技术背景真实**：故事要嵌入真实技术场景（生产环境崩盘、代码审查、架构决策等）

## Story Structure
```
开场 → 冲突升级 → 正面交锋 → 高潮对决 → 结局反转
```

## Conflict Escalation Pattern
- 第一个月：暗流涌动，言语试探
- 第二个月：正面冲突，拍桌子骂人
- 第三个月：全面战争，服务器崩盘，办公室对峙
- 结局：强势方胜出，弱势方走人或被教育

## Character Archetypes
- **主角**：技术总监/猫总监，严厉，直接，技术过硬，骂人不带脏字但句句致命
- **反派**：不懂技术还瞎指挥的CTO/领导，傲慢，被主角怼到走人
- **团队**：怕主角但敬主角，因为主角骂人是因为负责

## Tone Guidelines
- 短句，命令句，一句一个意思
- 对话要有火药味，不绕弯子
- 技术细节要真实（PostgreSQL vs NoSQL, 并发控制, 生产环境事故等）
- 结局要爽，主角赢，反派被教育

## Pitfalls
- ❌ 不要写轻微幽默或冷笑话，用户说不好笑
- ❌ 不要写温和的冲突，用户说不够激烈
- ❌ 不要让主角妥协或认输
- ✅ 冲突要动手骂人级别
- ✅ 技术细节要硬核
- ✅ 角色性格要符合用户画像（严厉将军风格）
